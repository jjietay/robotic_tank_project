Project: Robotic Tank (Pico W + Pi 4 + YD Lidar + USB Camera + ROS 2)

Repository layout
--------------------

- docs/
  - Hardware diagrams, reference images, and other documentation.
  - Example: wiring diagrams, board photos, reference PNGs.

- firmware/
  - Code that runs on the microcontroller (Raspberry Pi Pico / RP2040).

  - firmware/pico_motor_control/
    - Firmware project for motor control, built with CMake + Pico SDK.
    - You normally edit only the source and config files here.

    - Typical editable files:
      - CMakeLists.txt – firmware build configuration.
      - src/ (or similar) – C/C++ source files such as main.cpp, motor control
        logic, protocol handling, etc.
      - Any custom headers for the firmware.

    - Generated files:
      - build/ – CMake build directory (object files, libraries, ELF, UF2, etc.).
        This folder is ignored by Git and recreated by CMake.
        Do NOT edit files in build/ by hand.

- software/
  - Host‑side software; currently ROS 2 workspace for the rpi4.

  - software/ros2_ws/
    - ROS 2 workspace (Humble) for nodes that run on the Pi.

    - src/
      - All ROS 2 packages live here. Each subfolder is a separate package.

      - rc_car_teleop/ (Python ament_python package)
        - Teleoperation node for the RC car.

        - Files to know:

          - package.xml
            - ROS 2 package metadata (name, version, maintainer).
            - Declares dependencies (<exec_depend>rclpy</exec_depend>, etc.).
            - Edit when you add/remove dependencies or update maintainer info.

          - setup.py
            - Python packaging and entry point configuration.
            - Controls the console script used by “ros2 run”.
            - Important fields:
              - name / version / description / license.
              - entry_points["console_scripts"], e.g.
                teleop_node = rc_car_teleop.teleop_node:main
                - Left side (“teleop_node”) is the command name.
                - Right side points to main() in rc_car_teleop/teleop_node.py.
            - Edit when you change how the node is launched or add extra scripts.

          - setup.cfg
            - Install configuration for ament_python.
            - Typically:
              [develop]
              script_dir=$base/lib/rc_car_teleop
              [install]
              install_scripts=$base/lib/rc_car_teleop
            - Usually no need to change unless you rename the package.

          - resource/rc_car_teleop
            - Marker file used by the ROS package index.
            - Created once; normally never edited.

          - rc_car_teleop/__init__.py
            - Marks the directory as a Python package.
            - Usually empty; no changes needed.

          - rc_car_teleop/teleop_node.py
            - Main teleoperation node implementation.
            - Contains:
              - ROS 2 node subclass (e.g. TeleopNode).
              - Serial communication with the Pico (port, baud rate).
              - Keyboard or command logic for speed/steering.
              - def main(args=None): – the entry point used by setup.py.
            - This is the primary file you edit when changing behavior:
              - Serial port name (e.g. /dev/ttyACM0 vs /dev/ttyUSB0).
              - Command protocol to the Pico.
              - Mapping between inputs and motor commands.
              - Logging, parameters, etc.

          - test/
            - test_copyright.py
            - test_flake8.py
            - test_pep257.py
            - Style and license tests used by “colcon test”.
            - Normally unchanged unless you customize testing rules.

    - build/
      - Generated by “colcon build”.
      - Contains intermediate build artifacts.
      - Ignored by Git; safe to delete and regenerate.

    - install/
      - Output of “colcon build” (installed packages and launchable scripts).
      - “ros2 run” uses executables from here after sourcing install/setup.bash.
      - Ignored by Git; recreated on each build.

    - log/
      - Build and test logs produced by colcon.
      - Useful for debugging build issues.
      - Ignored by Git.


Typical workflow
-------------------

On Mac:
  1. Edit source (teleop_node.py, firmware source, etc.).
  2. Build/test locally if desired.
  3. git add changed files, git commit, git push.

On Raspberry Pi:
  1. git pull to get latest code.
  2. cd software/ros2_ws (or the workspace path).
  3. source /opt/ros/humble/setup.bash
  4. colcon build --packages-select rc_car_teleop
  5. source install/setup.bash
  6. ros2 run rc_car_teleop teleop_node
